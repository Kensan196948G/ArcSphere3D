# ADR-0003: JWT RS256 Asymmetric Keypair

## Status

Accepted

## Date

2026-05-16

## Context

ArcSphere3D の認証トークンは当初 HS256 (共有秘密鍵) で署名していた。
共有秘密鍵方式は以下の問題を持つ:

- 検証側 (例: BFF、マイクロサービス) も秘密鍵を保持する必要があり、鍵漏洩時の影響範囲が広い。
- 将来的に複数サービスでトークンを検証する場合、秘密鍵の安全な配布が運用上の負担になる。
- OAuth 2.0 / OIDC との統合では RS256 / ES256 が標準的に要求される。

`python-jose[cryptography]` は既にインストール済みであり、
`cryptography` ライブラリ経由で RSA 鍵生成・PEM シリアライズが利用可能。

## Decision

JWT 署名アルゴリズムを **RS256 (RSASSA-PKCS1-v1_5 + SHA-256)** に移行する。

- 署名は RSA 秘密鍵 (PEM) で行い、検証は公開鍵 (PEM) のみで行う。
- 本番環境: 環境変数 `JWT_PRIVATE_KEY_PEM` / `JWT_PUBLIC_KEY_PEM` で鍵を注入する。
  - 鍵長は最低 2048-bit RSA とする。
  - 本番環境 (`app_env` が `DEV_ENVS` 以外) で `JWT_PRIVATE_KEY_PEM` が空の場合、起動時に `ValueError` を raise してデプロイを拒否する。
- 開発 / テスト環境: 鍵が未設定の場合、起動時にエフェメラル RSA-2048 キーペアをメモリ内で生成し、プロセス内でキャッシュする。これにより CI 環境での追加設定不要を維持する。

### 鍵管理指針

| 環境 | 秘密鍵の保管場所 |
|------|-----------------|
| 本番 | AWS Secrets Manager または Kubernetes Secret (base64 PEM) |
| ステージング | 同上 (本番とは別鍵) |
| CI/CD | conftest.py がエフェメラル鍵を自動生成 |
| ローカル開発 | `.env` (`.gitignore` 済み) または未設定 (エフェメラル) |

## Consequences

### Positive

- 公開鍵のみを持つサービスがトークンを検証できる (秘密鍵の最小配布)。
- 将来の OIDC / JWKS 公開鍵エンドポイント追加が自然な拡張になる。
- HS256 の共有秘密鍵漏洩リスクを排除する。

### Negative / Trade-offs

- RS256 の署名・検証は HS256 より CPU コストが高い (数百マイクロ秒 vs 数十マイクロ秒)。
  アクセストークン TTL を長め (デフォルト 60 分) に保つことで署名頻度を抑える。
- 環境変数 `JWT_PRIVATE_KEY_PEM` / `JWT_PUBLIC_KEY_PEM` が新たな必須の運用設定となる。
  既存 `JWT_SECRET` の設定は不要になり、削除が必要。
- 鍵ローテーション時は旧公開鍵で発行済みトークンが TTL 切れするまで両鍵を並行運用する必要がある (今後の ADR で対応)。
